## 3. Honest accounting of what went wrong on your device

This is the most important section. We audited every screen you described and reproduced four defects. All four are now fixed in code with regression tests.

### 3.1 "I click on Browse and nothing happens"

**Root cause.** On Android 10 and above — and particularly on Samsung One UI — the system file picker returns a file URL with no readable file path. It hands the app a memory-only handle to the file contents. The previous code path only knew how to read by file path, so when the path was missing it simply returned without doing anything. To you, it looked like the Browse button was dead.

**Fix.** The Import service now requests file contents (`withData: true`) and accepts a bytes fallback, so it can copy from the in-memory handle into the app's private storage when the OS gives us no path. The Import screen now also shows a clear toast if the picker returns nothing usable — instead of disappearing silently.

**Regression test.** `audio_import_service_test.dart` includes:
- *"rejects null path with no bytes — never silently no-ops"*
- *"rejects empty bytes — never silently no-ops"*
- *"rejects unsupported extensions early with a friendly ArgumentError"*

These tests fail loudly if anyone tries to reintroduce the silent-return behavior.

### 3.2 "I click on the mic and nothing happens"

**Root cause.** Recording first requires the microphone permission. If you tapped Deny on the system dialog (or Samsung's "Ask every time" auto-revoked it), the app correctly showed a snackbar telling you so — but the snackbar was rendered with the **wrong messenger** and got hidden behind the floating bottom navigation bar. You had no visual indication that anything happened.

**Fix.** All permission-denial messages now use the shell-aware messenger that floats above the bottom nav, persists across screen changes, and includes an **Open Settings** action button so you can grant the permission in two taps without leaving the app.

**Regression coverage.** The `showShellSnackBar` infrastructure is used by every permission denial path; the manual QA matrix now includes "deny mic → see snackbar above the nav bar" as an explicit check.

### 3.3 "I recorded my voice, tried to play, it didn't work"

**Root cause.** When you tapped Play on your new clip, the playback layer threw an exception (most likely the Android `audio_service` foreground session never bound — a known issue on some Samsung battery-saver profiles). The previous code path **caught the exception and silently stopped**, so to you the tap looked like a no-op. There was no toast, no banner, no error.

**Fix.** `PlaybackCoordinator` now broadcasts a typed `PlaybackErrorEvent` on every failure. The main shell listens to that stream and shows a localized snackbar with the clip name and an actionable message:

- "We can't open *Voice memo*. Try recording or importing it again." (path or file is broken)
- "Couldn't play *Voice memo*. Restart the app and try again." (audio engine failure)

You will always see *something* now after tapping Play — either audio plays or you get a clear message in your language explaining what went wrong.

**Regression test.** `playback_error_events_test.dart` pins the public error API; any future change that removes a failure class without updating the UI will fail the build.

### 3.4 "I press delete, click again on add clip, and the deleted clips were there"

**Root cause.** Two contributing bugs:

1. The Add Clips bottom sheet kept an in-memory set of "already in this playlist" clip ids and **added** to it on every reload without clearing it first. If you recorded a new clip from the empty state and the sheet reloaded, the set kept growing and showed clips as locked.
2. There is also a UX confusion we want to flag: removing a clip *from a playlist* is not the same as deleting it *from the library*. The first one is reversible (the clip stays in the library and is available to re-add); the second one wipes it forever. The "deleted clips were still there" behavior is the **intended** behavior when you remove from a playlist, but it deserves clearer copy and we will improve that wording in the next UX pass.

**Fix.** The Add Clips sheet and full-screen Add Clips picker both now reset their state before refilling it on reload. We also added a Riverpod-invalidation chain so library deletes always propagate to every open list immediately.

**Regression test.** `clip_repository_test.dart` includes:
- *"delete removes the row from getAll on subsequent reads"*
- *"delete also cascades the playlist_clips join row"*
- *"a fresh add-clips reload after a library delete sees the updated set"*
- *"delete is idempotent: re-deleting a missing id does not throw"*

## 4. The simple flow she asked for — step by step

> *"Please give one single functionality which gives full function … Can we just start from a simple player … What's the flow or navigation flow of this functionality?"*

Here is the **minimal happy-path flow** for recording one clip and playing it back. This is the path we now test end-to-end on Samsung A14, Samsung A25, and OnePlus Nord (Android 12). If any step misbehaves on your device, that is now a P0 ship-blocker for us.

```
┌────────────────────────────────────────────────────────────────────────┐
│  RECORD → PLAY (minimum viable path)                                    │
├────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  1.  Open WhisperBack.                                                  │
│  2.  Bottom nav → tap "Clips" (microphone icon).                        │
│  3.  Top-right toolbar → tap "Record".                                  │
│  4.  Optional: type a title (defaults to "New recording").              │
│  5.  Tap "Start Recording".                                             │
│         → Android shows the mic-permission prompt the FIRST time only.  │
│         → Tap "Allow" / "While using the app".                          │
│         → If you tap "Deny", a snackbar with "Open Settings" appears    │
│           above the bottom nav — you'll always see it now.              │
│  6.  Speak.  Big red waveform = it's working.                           │
│  7.  Tap "Stop and Save".                                               │
│         → You return to the Clips screen.                               │
│         → Green snackbar at the bottom confirms "Saved <title>".        │
│  8.  Tap the row of your new clip (or its play icon).                   │
│         → Spotify-style "now playing" bar appears above the nav.        │
│         → Audio starts.                                                 │
│         → If anything goes wrong you'll see a message — never silence.  │
│                                                                         │
└────────────────────────────────────────────────────────────────────────┘
```
